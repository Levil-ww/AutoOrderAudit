修复一、
修复adapter的商品行构建问题： 1.备注中含有两个商品，但实际只输出了一个，并且默认第二个
    问题出在 adapter.py 的 update_merchant_code 方法中，构建 API 请求体 orderItems 的逻辑有误。
         索引	                      逻辑	                                         id 字段值	      结果
        idx=0	idx < len(order.items) → 使用 order.items[0]	                        item_001	  ✅ 更新第1行
        idx=1	idx >= len(order.items) → 回退到 template_item (= order.items[0])	也是 item_001	❌ ID重复！

两个商品行发送了 完全相同的 id、sysOid、oid，方果ERP 将它们视为同一行，第二个（45×90cm）覆盖了第一个（25×57cm），所以界面上只能看到45×90cm那一行。

方案：在 adapter.py 的 update_merchant_code 中，对于超出现有商品行数量的新建商品行，使用 空白 OrderItem（ID字段为空），让ERP API识别为"新建行"而非"更新现有行"：
    索引	        id	            含义
    idx=0	item_001	更新原有商品行为 25×57cm
    idx=1	""(空)	    ERP 自动为 45×90cm 创建新行

测试结果：
    解析层正确返回2个商品行 ✓
    第1行保留原商品行ID → ERP更新该行 ✓
    第2行使用空白ID → ERP创建新行，不会再冲突 ✓

虽然能够构建新的商品行，但是如果订单里含有赠品行，构建的新的商品行会自动占用赠品行 - 修复新建商品行占用赠品行的问题
    问题：我之前用 OrderItem(order_id=order.trade_id) 创建空白行（id=""），ERP API 可能将空 ID 错误匹配到现有赠品行。

    修复：改用 _build_default_item(order, p)，使用 order.trade_id 作为占位 ID（不是真实商品行 ID），ERP 会识别为新建行，不会误匹配任何现有行。

修复三、
在运行程序时，遇到已经修改过的商品编码，它不会再去检验是否正确，无法确定第一次修改的是否正确，再次运行程序的新编码不会覆盖原有的编码
修复 - 增加二次运行编码正确性校验
    问题：无论编码是否已修改正确，每次运行都会重新调用 API 修改。

    修复：在 API 调用前，对比当前订单的商品编码与解析出的期望编码。如果所有期望编码都已存在，则跳过修改：
        场景	        当前编码	        期望编码	        行为
        首次运行	    旧编码	        {A, B}	    ❌ 不匹配 → 修改 ✓
        二次运行	    {A, B}	        {A, B}	    ✅ 已正确 → 跳过 ✓
        手动改错了   {A, 错码}	        {A, B}	    ❌ 不匹配 → 修改 ✓

修复五、
商品行位置顺序：之前新建的商品行(45x90)放到了数组位置[1]，正好是赠品行原来的位置，ERP按位置匹配时把赠品覆盖了。
新逻辑 --
    原订单:  [商品(位置0)] [赠品(位置1)]
旧逻辑（错误）:
  [0] = 更新后的商品(25x57)  ← 更新位置0 ✅
  [1] = 新建商品(45x90)      ← 覆盖位置1！赠品没了 ❌

新逻辑（正确）:
  [0] = 更新后的商品(25x57)  ← 更新位置0 ✅
  [1] = 保留的赠品(不变)      ← 保持位置1不变 ✅
  [2] = 新建商品(45x90)      ← 追加到末尾，真正的新建行 ✅
  用 order_items_map 按原索引存储，新建行通过 new_items_to_append 统一追加到最后，确保赠品行永远保留在原来的位置
