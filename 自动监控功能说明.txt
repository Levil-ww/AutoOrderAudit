需求分析 -- 只要有待处理订单就自动审单  -  属于持续监控场景，更适合“轮询监控模式”

一、轮询监控模式 与 定时任务 的方案对比
        对比项	    定时任务（cron）	                轮询监控模式
        适用场景	    固定时间点执行（如每天凌晨2点）	    持续运行、有新单就处理
        实时性	    差（取决于间隔）	                好（可设5-30秒轮询）
        实现复杂度	中	                                低
        资源占用	    每次启动都要初始化	                一次初始化，持续运行
        与GUI集成	差（通常是独立进程）	            好（同一进程内线程）

二、 核心思路分析
    GUI 主线程 ←→ 监控线程（独立 daemon 线程）
                    ↓
             循环：sleep → 查询订单 → 有单就审 → 无单继续等

四、关键注意事项
    4.1. Token 过期处理
    监控过程中 Token 可能过期，需要：
        每次查询前检查 Token 有效性
        失效时自动暂停监控并弹窗提示登录

    4.2. 避免重复处理
    同一批订单可能被多次轮询到，需要：
        利用方果ERP的状态机制（审单后状态会变化，不会再出现在待整理列表中）
        如果方果有这个机制就不用担心（从代码看 query_status=1 应该就是查待整理状态）

    4.3. 并发性控制
        审单进行中时，下一轮轮询应该跳过，避免重复执行
        用 _running 标志位控制（你代码里已经有了）

    4.4. API 调用频率限制
        轮询间隔不要太短（建议最短10秒）
        避免频繁调用给方果服务器造成压力
        可以设置为：有订单时处理快些，没订单时间隔拉长（自适应间隔）

    2.gui.py -- 新增洗的监控功能

        退出处理	退出时优雅停止监控线程

6.2 自适应间隔算法
    有订单 → 10秒后再次检查（快速响应）
    无订单 → 30s → 45s → 67s → 101s → 151s → 180s（封顶）
    发现新订单 → 重置为10秒

6.3 使用流程
    1.打开程序 → 检测到已登录 → 弹窗「是否开启自动监控？」
    2.点击「确定」 → 监控线程启动，开始轮询
    3.检测到待处理订单 → 自动执行审单 → 10秒后再次检查
    4.无订单 → 逐步拉长间隔（最多3分钟），减少API调用
    5.点击「⏸ 停止监控」 → 随时关闭监控

6.4 安全保障
    1.Token 失效时自动暂停监控并提示重新登录
    2.审单进行中不会重复触发（_running 标志位互斥）
    3.手动审单与自动监控互不冲突
    4.等待期间可被随时中断（Event.wait() 代替 sleep()）
    5.退出程序时优雅停止监控线程

6.5 优化自动监控
    1. 修改内容：在自动监控模式下，将 confirm_callback 设置为 None，这样审单时不会再弹出确认弹窗，直接自动处理订单。
    2. 效果：
        ✅ 自动监控模式：检测到待处理订单后，直接自动审单，无需点击确认
        ✅ 手动审单模式：点击「开始审单」时，仍然会弹出确认对话框（保持原有行为）
    现在流程更加顺畅：打开程序 → 点击「确定」开启自动监控 → 检测到订单自动处理 → 无需任何人工干预。
